# ORCA Digital Twin Implementation Guide

**Project:** ORCA – Onboard Resource Conflict Arbitrator

**Document Version:** 1.0

**Purpose:** Software Implementation Guide

---

# 1. Overview

The ORCA Digital Twin is implemented as a collection of independent Python modules.

Each module represents a major spacecraft subsystem and is responsible for maintaining its own internal state while exposing telemetry through a common interface.

The simulator coordinates all subsystem updates at a fixed **1 Hz** execution rate.

# 4. Module Responsibilities

## config.py

Stores all configurable simulation constants.

Examples:

* Battery Capacity
* Orbit Period
* Solar Array Power
* Thermal Limits
* RF Thresholds

No simulation logic exists in this file.

---

## battery_model.py

Responsible for spacecraft energy storage.

Maintains:

* State of Charge
* Battery Voltage
* Battery Current
* Available Energy

Updates battery state from net electrical power.

---

## solar_model.py

Calculates electrical power generated by the solar arrays.

Uses:

* Orbit Time
* Orbit Phase

Produces:

* Solar Power
* Sunlight/Eclipse State

---

## payload_model.py

Represents spacecraft payload operation.

Maintains operating states for:

* PNT
* SatCom
* Earth Observation
* Edge AI

Calculates total payload power consumption.

---

## power_bus.py

Represents the spacecraft electrical distribution system.

Combines:

* Solar Generation
* Payload Demand

Computes:

* Net Power
* Battery Charge / Discharge State
* Bus Status

---

## thermal_model.py

Models spacecraft temperature using a first-order lumped thermal model.

Calculates:

* Heat Generation
* Passive Cooling
* Temperature
* Thermal Status

---

## rf_model.py

Estimates RF coexistence between active payloads.

Uses a configurable conflict matrix to compute:

* RF Margin
* RF Status
* Active RF Conflicts

---

## telemetry.py

Collects telemetry from every subsystem.

Produces a unified telemetry packet for ORCA.

---

## simulator.py

Coordinates every subsystem.

Responsible for:

* Simulation Clock
* Update Order
* Mission Events
* Telemetry Generation

This is the entry point of the Digital Twin.

# 6. Data Flow

Subsystem outputs become inputs for downstream models.

Example:

```text
Solar Power
        │
        ▼
Power Bus
        │
        ▼
Battery
```

Another example:

```text
Payload States
        │
        ▼
RF Model
```

And:

```text
Payload Power
        │
        ▼
Thermal Model
```

---

# 7. Telemetry Interface

Each subsystem exposes its current state through a public method.

Typical interface:

```python
get_state()
```

The telemetry generator combines all subsystem outputs into a single packet that is consumed by ORCA.

---

# 8. Coding Principles

The Digital Twin follows several software engineering principles:

* Modular subsystem design
* Object-oriented architecture
* Single responsibility per module
* Configurable constants
* Fixed-step deterministic execution
* No shared global state
* Clear separation between simulation and decision logic

---

# 9. Extending the Digital Twin

Additional spacecraft subsystems can be added without modifying existing modules.

Examples:

* Attitude Control
* Orbit Propagation
* Communication Links
* Radiation Effects
* Propulsion
* Fault Detection
* Sensor Models

Each new subsystem should expose telemetry through the same interface used by existing modules.

---

# 10. Relationship to ORCA

The Digital Twin does not make decisions.

Its only responsibility is to simulate spacecraft behaviour and generate telemetry.

The ORCA Decision Engine will consume this telemetry to:

* Schedule payloads
* Resolve resource conflicts
* Protect spacecraft safety
* Maximize mission utility

This separation allows the Digital Twin to remain reusable regardless of the AI algorithm selected in future development.
